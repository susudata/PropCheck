# ADR-002: Kompresja zdjęć do zaoszczędzenia localStorage

## Status
✅ Zaakceptowana (obowiązująca)

## Data
2026-05-11

## Problem
Zdjęcia niekompresowane zajmują 5-100MB per plik, szybko przepełniając localStorage (limit 5-10MB). Użytkownicy nie mogą dodać więcej niż 1-2 zdjęć bez błędu "QuotaExceededError".

## Rozwiązania rozpatrywane

### 1. Nie kompresować (status quo)
**Zalety:**
- Brak dodatkowego przetwarzania
- Pełna jakość zdjęć

**Wady:**
- Aplikacja crash'a po 1-2 zdjęciach
- Zła user experience
- Brak realnego MVP

### 2. Kompresja server-side (Supabase)
**Zalety:**
- Bezpieczne, niezawodne
- Profesjonalny CDN

**Wady:**
- Wymaga backendu
- Delayed (asynchroniczny)

### 3. Kompresja client-side (Canvas API)
**Zalety:**
- Natywny JavaScript, bez bibliotek
- Instant (synchroniczny)
- Oszczędza traffic
- Idealne dla MVP

**Wady:**
- Wymaga implementacji
- Zmienna jakoość w zależności od urządzenia

## Decyzja
**Kompresja client-side przy użyciu Canvas API**

### Implementacja

```javascript
function compressImageToBase64(file, maxWidth, maxHeight, quality) {
  return new Promise((resolve, reject) => {
    const reader = new FileReader();
    reader.onload = (e) => {
      const img = new Image();
      img.onload = () => {
        // Oblicz nowe wymiary (aspect ratio)
        const ratio = Math.min(maxWidth / img.width, maxHeight / img.height, 1);
        const canvas = document.createElement('canvas');
        canvas.width = img.width * ratio;
        canvas.height = img.height * ratio;
        
        // Rysuj i kompresuj
        const ctx = canvas.getContext('2d');
        ctx.drawImage(img, 0, 0, canvas.width, canvas.height);
        
        // Konwertuj do JPEG (lepsze compression niż PNG)
        const compressed = canvas.toDataURL('image/jpeg', quality);
        const ratio_pct = ((compressed.length / e.target.result.length) * 100).toFixed(1);
        console.log(`Image compressed: ${(e.target.result.length/1024).toFixed(0)}KB → ${(compressed.length/1024).toFixed(0)}KB (ratio: ${ratio_pct}%)`);
        
        resolve(compressed);
      };
      img.src = e.target.result;
    };
    reader.readAsDataURL(file);
  });
}
```

### Parametry kompresji

| Typ zdjęcia | Max Width | Max Height | Quality | Cel |
|---|---|---|---|---|
| Issue photo | 800px | 800px | 0.6 | ~90% redukcja, wystarczająca klarowność |
| Floorplan | 1200px | 1200px | 0.7 | ~85% redukcja, lepszej szczegółów |
| Demo data | 1200px | 1200px | 0.7 | Zmniejsza rozmiar initial data |

## Konsekwencje

### Pozytywne
- Users mogą teraz dodać 8-10 zdjęć per issue
- Brak "QuotaExceededError"
- Szybko i niezawodnie
- Zero zależności

### Negatywne
- Utrata jakości obrazu (0.6 quality)
- Konwersja PNG → JPEG (brak transparencji)
- Brak metadanych EXIF (mniej istotne dla tego use case)

## Performance Impact
- **Client:** Asynchroniczny process, minimal impact (~100ms per image)
- **Network:** Oszczędzenia (jeśli kiedyś będzie backend)
- **Storage:** 90% redukcji = drastyczne zmniejszenie quotaExceededError

## Testowanie
1. Dodaj 5+ zdjęć do usterki
2. Sprawdź DevTools Console na compression ratios
3. Sprawdź localStorage size (DevTools → Storage)
4. Edytuj issue i dodaj więcej zdjęć (nie powinno crash'ować)
