# ADR-001: localStorage jako główny storage dla MVP

## Status
✅ Zaakceptowana (obowiązująca)

## Data
2026-05-11

## Problem
W fazie MVP aplikacja musi działać szybko bez zależności od backendu. Potrzebny jest mechanizm przechowywania danych użytkownika (nieruchomości, usterki, zdjęcia).

## Rozwiązania rozpatrywane

### 1. localStorage
**Zalety:**
- Natywne wsparcie we wszystkich przeglądarkach
- Synchroniczny dostęp, brak latencji
- Prosty JSON.stringify/parse
- Idealny do prototypowania

**Wady:**
- Limit ~5-10MB (problematyczne przy wielu zdjęciach)
- Brak synchronizacji między urządzeniami
- Brak szyfrowania
- Brak versionowania/migracji

### 2. IndexedDB
**Zalety:**
- Nieograniczona pojemność (~50MB+)
- Asynchroniczny
- Wsparcie dla blob objects
- Lepszy A/B testing

**Wady:**
- Bardziej skomplikowane API
- Kurba uczenia się
- Overkill na MVP

### 3. Firebase Realtime Database
**Zalety:**
- Cloud storage (sync między urządzeniami)
- Real-time updates
- Autoryzacja

**Wady:**
- Dodatkowa zależność
- Brak offline-first

### 4. Supabase PostgreSQL
**Zalety:**
- Relacyjne dane
- Skalowalne
- Integracja z Auth
- Sugerowane w AGENTS.md

**Wady:**
- Wymaga backendu
- Kompleksowa konfiguracja
- Overkill na MVP

## Decyzja
**Wybrali localStorage dla MVP z planem migracji na Supabase Phase 2**

### Implementacja
```javascript
// Zapis
const issues = [{ id: "iss-1", name: "Cieknący kran", ... }];
localStorage.setItem('propcheck_issues', JSON.stringify(issues));

// Odczyt
const issues = JSON.parse(localStorage.getItem('propcheck_issues') || '[]');

// Error handling
try {
  localStorage.setItem('propcheck_issues', JSON.stringify(issues));
} catch (QuotaExceededError) {
  // Fallback: zapis bez zdjęć
  alert('Storage pełny. Zdjęcia nie zostały zapisane.');
}
```

### Mitigation dla limitu 5-10MB
1. **Kompresja zdjęć:** 800x800px, quality 0.6 → ~90% redukcja
2. **IndexedDB dla zdjęć:** Zaplanowana w Phase 2
3. **Service Worker:** Caching, offline-first
4. **Error handling:** Graceful degradation (zapis bez zdjęć)

## Konsekwencje

### Pozytywne
- Szybkie prototypowanie
- Zero zależności backendu
- Natywne wsparcie przeglądarki
- Idealne do testowania UX

### Negatywne
- Brak synchronizacji między urządzeniami
- Limit pojemności (rozwiązany kompresją)
- Brak bezpieczeństwa (zaplanowana migracja)

## Plan migracji
Jeśli limit localStorage będzie problematyczny przed Phase 2:
1. Włączyć IndexedDB dla zdjęć
2. Zachować localStorage dla metadanych
3. Hybrid storage model
